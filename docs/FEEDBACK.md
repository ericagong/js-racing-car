    [ ] state나 callbacks 등의 namespace로 묶기

-   이상이라면, 객체 분리의 순간이 아닌지 고민하기(하나 이상의 역할을 하고 있을 가능성이 높음)
    [ ] Car 객체에서 유효성 검사 책임 Name 객체로 분리
    [ ] Cars 객체를 역할에 따라 분해

    -   매 Round 플레이: cars랑 게임 전진 조건 받아 플레이 후 결과 반환

    [ ] getter 도입해 메소드 수 줄이기

-   역할 위임 잘하기
    [ ] 2 level 이상 들어가서 일처리 하지 않고, 명령하기

2. 가독성

2.1 네이밍 컨벤션

-   파일명
    [V] export default 파일명 일치
-   함수명
    [V] 객체 내부 함수 네이밍 시, 모델명 포함 X
    [ ] 단, 외부에서 불러다 사용할 때 모델명 붙여주기
-   상수 네이밍
    [V] 상수값이 아니라 도메인 내 상수 역할을 이름으로 사용

2.2 테스트 코드

-   유사한 test code는 describe로 그룹화하거나 병합
-   test 내 들어가는 input과 결과를 한 눈에 보여지도
-   test description을 추상적이지 않고 명확하게 작성: 처음 테스트 코드 읽는 사람이 맥락 모르고도 바로 알게끔!
