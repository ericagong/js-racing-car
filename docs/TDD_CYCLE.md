# 직접 정의한 나만의 TDD Cycle

## 1. 전체 문제가 해결되었을 때, 상태를 `한 문장`으로 정의

-   **목표 상태**가 무엇인지 명확히 하기 위해 전체 문제 해결 시의 상태를 **한 문장**으로 표현한다.

---

## 2. 문제를 적당한 난이도로 기능 단위로 분리하기

-   **1에서 정의한 한 문장**에서 다음을 추출한다:

    1. **가장 작은 버전**: 기능이 구현될 수 있는 최소 단위
    2. **핵심 기능**: 복잡성을 제거하고 남는 주요 기능

-   **핵심 기능 구현 순서**:

    -   먼저 **가장 작은 버전의 핵심 기능부터** 구현한다.
    -   이후, **복잡성을 늘려가며 디테일**을 잡는다.

-   **복잡성 줄이는 방법**:

    -   `UI 로직`과 `도메인 로직`을 분리하여 각각 관리한다.

-   **TDD 사이클 종료**:
    -   전체 기능이 구현되어 **1의 상태가 되면 사이클을 중단**한다.

---

## 3. 지금 할 수 있는 기능을 선택하기

-   2에서 뽑아낸 기능 중 **현재 구현 가능한 기능**을 선택한다.

### **선택 기준**

✅ 해당 기능을 구현하면 향후 구현이 더 편해지는가?  
✅ 해당 기능이 **1의 문장 상태**에 더 가까워지는가?

---

## 4. 선택한 기능을 구체화하고 시뮬레이션하기

-   선택 기능 내부의 **세부 기능**을 bullet point로 작성한다.

    -   이때, **확장성은 고려하지 말고 빠른 구현에 초점**을 맞춘다.
    -   확장성은 리팩토링 과정에서 처리한다.

-   **edge case**를 뽑아내고 **check point**로 작성한다.

---

## 5. 실패하는 테스트 코드 작성 후 구현하기

-   **세부 기능 단위**로 **실패하는 테스트 코드**를 작성한다.
-   테스트를 통과하도록 코드를 작성한다.

-   **기능 완료 시**:
    -   선택한 기능의 모든 세부 기능을 구현하면 **6번 단계**로 이동한다.

---

## 6. 기능 단위로 리팩토링 진행하기

-   리팩토링 시 다음 네 가지를 고려한다:
    1. **중복 제거**
    2. **가독성 개선**
    3. **확장성 확보**
    4. **성능 최적화**

### **리팩토링 시 고려사항**

1. **중복 제거**:

    - **모듈화가 필요한 부분**을 식별하고 처리한다.

2. **가독성 개선**:

    - 코드가 **의도를 명확히 드러내는지** 점검한다.
    - **목적에 맞는 메소드/함수**를 사용했는지 확인한다.

3. **확장성 확보**:
    - **요구사항 변경 가능 지점**을 체크하고 외부로 뽑아낸다.

---

## 7. 사이클 반복하기

-   2번 또는 3번 단계로 돌아가 **사이클을 반복**한다.
