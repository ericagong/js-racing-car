## 1. 나만의 요구 사항 분석 사이클 정의

-   [나만의 TDD Cycle](./TDD_CYCLE.md)에서 확장하여 나만의 **RDD(Role-Driven Design) 사이클**을 정의하였습니다.
-   💡 **리팩토링 시 고려사항**: `가독성`, `재사용성`, `성능`

### 요구 사항 분석 사이클

1. **요구사항 기반으로 (1) 역할, (2) 객체, (3) 협력관계 추출**

    - 도메인 로직과 UI 로직을 **분리**
    - **도메인 로직**부터 구현하고, 이후 **UI 로직** 진행

2. **객체 추출 후 객체 관계도 작성**

    - 객체의 **역할과 책임이 하나인지** 확인

3. **객체 관계도를 통해 전체 프로그램의 흐름 파악**

    - 외부에서 **주입되어야 할 조건**을 파악

4. **Controller 추가**

    - 역할에 따른 책임 분리

5. **도메인 로직 내 객체별 역할 및 필드 추출**

    - 역할이 여러 개이거나 필드가 너무 많다면 **객체 분리**
    - 객체 간 협력에 필요한 메소드는 **public 필드**로 설정

6. **사용 목적에 따른 설계 방식 선택**

    - **Class** vs **IIFE 모듈**

7. **구현 시작**

## 2. 객체가 하나의 책임만 가지도록 하는 방법

-   **객체 책임을 잘 분리하는 방법**:
    -   **객체 분리를 고민해야 하는 시점(Trigger 요소)**:
        -   객체에 **4개 이상의 상태**가 존재할 때
        -   **지나치게 많은 메소드**가 포함될 때

## 3. 패턴 도입 의사 결정 시 주의 사항

-   💡 **패턴은 문제 해결을 위한 과정**일 뿐, **결과 자체가 되어서는 안 된다.**
-   불필요한 패턴에 **집착하지 말고**, 필요한 경우에만 도입해야 함

## 4. 테스트 코드 작성 시 주의 사항

### **테스트 코드 목적**:

1. **사이드 이펙트 발생 여부**를 검증
2. **엣지 케이스**를 잘 커버하는지 확인

### **불필요한 테스트 코드 지양**:

-   테스트를 위한 테스트 작성 **지양**
-   **커버리지 100%**에 집착하지 않기

### **테스트 대상**:

-   **핵심 로직과 public 메소드**만 테스트
-   만약 **private 메소드**가 핵심 로직을 담당한다면:
    -   해당 메소드를 사용하는 **public 메소드**를 검증
    -   이는 **잘못된 설계**일 가능성이 크므로, **리팩토링 시 재설계** 고려

💡 **중복된 테스트가 있다면**, 잘못된 설계를 했을 가능성이 높음

### **테스트 코드 작성 시 가독성 우선 고려**:

-   **변수화 지양**
-   **코드 효율성보다 가독성**을 우선 고려
