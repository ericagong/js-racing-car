### 1. 객체가 하나의 책임만 가질 수 있도록 분리

#### **_[As-Is] 하나의 객체가 여러 책임을 가짐_**

-   단일 객체가 여러 책임을 가지면 코드가 복잡해져 다음과 같은 문제점이 발생합니다:
    1. **변경 파급 범위 예측 어려움**: 코드 변경 시 어디까지 영향이 미치는지 파악하기 어려움
    2. **유지보수와 디버깅 어려움**: 책임 분리가 불명확해 문제가 발생했을 때 원인을 찾기 힘듦

#### **_[To-Be] 객체의 책임을 명확히 분리_**

-   **변경 지점:**
    -   ✅ `Car Model`에서 자동차 이름 유효성 검사 로직을 `CarName Model`로 분리
    -   ✅ 자동차 이름 배열의 **중복 여부 검사**와 **Car 배열 관리 책임**을 별도 파일로 분리
    -   ✅ 각 라운드의 자동차 결과 저장 책임을 **Round 객체**로 분리

### 2. 게임 이동 조건 관리 책임을 MoveStrategy로 일임 및 주입 지점 통일

#### **_[Before] 불필요한 전략 패턴 도입으로 인한 복잡성 증가_**

-   초기 설계에서는 자동차 전진 조건의 변경 가능성을 고려해 **전략 패턴**을 도입했습니다.

    -   추상 클래스인 `MoveStrategy`에서 두 가지 추상 메소드를 강제 구현:
        1. `isMovable()`: 이동 가능 여부 판단
        2. `generateNumber()`: 이동 여부를 결정하는 랜덤 숫자 생성

-   하지만 다음과 같은 문제점이 있었습니다:
    1. 이동 전략 관련 상수(`CAR_MOVE_STEP`)가 **constants.js** 등 여러 파일에 분산 관리됨
    2. `generateNumber()`가 존재해 이동 전략이 **숫자 기반**으로만 한정됨
    3. 추상 클래스 구조로 인해 **코드가 복잡해 이해가 어려움**
    4. 이동 전략의 **외부 주입 코드가 불명확**하여 책임이 불분명함

#### **_[After] MoveStrategy에서 이동 조건 통합 관리 및 주입 지점 통일_**

-   **변경 지점:**

    -   ✅ `MoveStrategy`에서 이동 조건과 관련된 모든 로직 통합 관리  
        (이동 여부 판단 함수, 인자 생성 함수, 이동 거리 포함)
    -   ✅ 추상 클래스에서 **구체 클래스**로 변경해 간결화하고 `new` 키워드로 인스턴스 생성
    -   ✅ **Car 인스턴스**에서 이동 시 **주입 받은 MoveStrategy 인스턴스** 참조

    ```javascript
    // Car.js
    export default class Car {
        #position;

        #move(step) {
            this.#position += step;
        }

        tryMove(moveStrategy) {
            if (moveStrategy.isMovable()) {
                this.#move(moveStrategy.step);
            }
        }
    }
    ```

    -   ✅ `MoveStrategy` 주입 지점을 **RacingGame constructor**로 제한하여 일관성 유지

### 3. MVC 패턴에 따라 Error 객체 분리 및 에러 처리를 위한 공통 인터페이스 추가

#### **_[Before] MVC 패턴에 따라 Error의 영역 분리가 이뤄지지 않음_**

-   이전 코드에서는, 어플리케이션 레벨의 `src/RuntimeError.js`를 `Model/ValidationError.js`에서 참조했습니다.
-   이 때, MVC 패턴을 사용하였음에도 불구하고, `View/*` 에서 에러가 발생한 경우 `Model/ValidationError.js` 모듈을 사용하였습니다.
-   **MVC 패턴에 따른 영역 분리**가 지켜지지 않아, 아래와 같은 문제점이 발생했습니다:
    -   `Controller`의 최상위 try-catch문에서 에러가 발생했을 때, 에러 발생 지점이 Model인지 View인지 특정할 수 없음
    -   **영역 혼재**로 **유지보수와 디버깅이 어려움**

#### **_[After] Error 객체 분리 및 공통 에러 처리를 위한 인터페이스 구현_**

-   **변경 지점:**
    -   ✅ `RuntimeError`를 상속하여 **View와 Model 각각 ValidationError 객체 구현해 별도 관리**
    -   ✅ `RuntimeError`에 `error.type` 인터페이스 추가해 Controller에서 공통 방식으로 에러 처리
        -   Model의 `ValidationError`와 View의 `ValidationError`에 **서로 다른 type**을 부여
        -   Controller의 `try-catch` 블록에서 `error.type`으로 에러 발생 위치를 명확하게 파악 가능
    -   ✅ Controlelr에서 **RuntimeError가 아닌 경우**, 예상치 못한 에러로 간주해 **프로그램 종료 처리**

### 4. RacingGame 내부 함수의 실행 순서 보장을 위한 state 변수 도입

#### **_[Before] 비순차적 호출로 인한 결과 보장 불가_**

-   이전 코드에서는, RacingGame에서 `set -> play -> getResult` 순으로 **순차적 실행**이 필요했습니다.
-   하지만 이를 **강제하는 수단**이 없어, 비순차적으로 호출될 경우 **원하는 결과를 보장할 수 없다**는 문제점이 있었습니다.

#### **_[After] state 변수 도입으로 순서 보장 및 에러 처리_**

-   **변경 지점:**
    -   ✅ RacingGame에 `state` 변수를 추가하고 **비순차적 호출 시 `InvalidCallOrder` 에러**를 발생
    -   ✅ RacingGame을 일반 함수 내보내기 모듈화 방식에서 class 모듈화 방식으로 변경

### 3. 테스트 코드 작성 목적을 고려한 테스트 코드 리팩토링

#### **_[Before] 목적을 잊은 테스트 코드 작성_**

-   기존 테스트 코드에서 다음과 같은 문제점이 있었습니다:
    1. 테스트 커버리지 100%에 집착하여 불필요한 테스트 작성
        - 과도한 `__tests__/Fixtures/*` 사용으로 가독성 저하
    2. `beforeEach`, `afterAll` 사용으로 코드 효율성을 추구했으나,  
       **가독성 저하**와 테스트 변경 시 예기치 못한 사이드 이펙트 발생

#### **_[After] 테스트 목적을 고려한 리팩토링_**

-   **변경 지점:**

    -   ✅ 불필요한 테스트 코드를 삭제하고, **테스트 작성의 핵심 목적**을 반영:

        1. **edge case** 감지
        2. **코드 변경 시 side effect** 감지

    -   ✅ 코드 효율성보다 **가독성을 우선**으로 고려하여 작성:
        -   **불필요한 변수화 지양**하여 테스트 케이스 바로 `it`에 작성
        -   **`beforeEach` 등 지양** 해 테스트 코드, 변경 시 네임스페이스로 인한 예기치 못한 부작용 방지

### ❓ MVC 패턴에 따른 파일 분리

-   MVC 패턴을 적용했는데, Model과 Controller, View의 하위 파일 중, 어디 위치 시킬지 고민되는 부분들이 있어 질문드립니다. 피드백 부탁드립니다.

1. Models/RacingGame.js에서 Controller에서 Model과 상호작용해야하는 부분들을 추상화해서 내보내고 있다는 느낌이 듭니다.

2. Controller에서의 CarNames, TotalRound 파일이 결국 사용자 입력값의 Validation을 담당하니 View에 가는게 적절하지 않을까 고민됩니다.

    - 특히 CarNames의 경우 중복 검사를 실시하는 것이 Model의 책임이 될 수도 View의 책임이 될 수 도 있다고 생각하는데요. 양쪽에서 검사하려면 어떤 방식이 좋을까요?

3. 관심사에 따라 분리한다면 DefaultMoveStrategy는 아예 Controller 폴더의 밖에 위치해야하는 것은 아닐까하는 생각이 듭니다.

### ❓ 순환 참조 이슈

-   코딩 컨벤션에 따라, 변수 선언 위치를 `constants.js`가 아니라 관심사에 가깝게 위치시키기 위해 각 Model 내부로 이동시켰습니다.
-   그 결과, `errors.js`에서 Error 객체의 message에 사용하기 위해 해당 변수를 불러올 때, 순환참조 이슈가 발생하더라구요. 그래서 우선은 Error 파일에서 이를 참조하지 않고 고정값을 사용하게 해두었습니다.
